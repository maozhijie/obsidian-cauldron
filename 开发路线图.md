# 丹道修炼系统 · 开发路线图

> 道生一，一生二，二生三，三生万物。
> 系统亦如修行——从最初的一炉丹火，渐次演化为完整的修仙世界。

## 一、已完成回顾

### v0.1.0 · 核心闭环（M1-M3）

v0.1.0 实现了从"采药"到"开炉"的最小可用闭环，完成了以下功能：

#### M1 · 核心数据采集
- 领域标签系统：创建、编辑、删除，每个标签绑定性味（神识/根骨/灵动/融合）
- Todo 完成事件监听与标签解析
- 药材采集：数量折算、稀有度判定（良材/珍材）、按日存储
- 药引采集：守时引/延时引（基于任务截止时间判定）

#### M2 · 番茄钟集成
- 内置简易番茄钟计时器
- 番茄钟完成 → 专注引采集
- 番茄钟中断 → 断念引采集

#### M3 · 封炉与炼丹
- 每日周期管理（用户自定义封炉时间）
- 君药/臣药判定算法
- 品级计算（凡品/灵品/宝品/神品）与纯度计算
- 丹药自动命名与记录生成
- 开炉界面展示（丹药详情、关键数据一览）

### v0.2.0 · 体系成形（M4-M5 + 外丹/内丹/统计）

v0.2.0 是一次全面的体系扩展与架构重塑，从单一炼丹闭环演化为完整的修仙世界。

#### 基础架构重构
- 类型系统扩展至 ~254 行，新增 20+ 接口（投注/播种/外丹/内丹/统计）
- VaultDataManager 拆分为 Facade + 3 子模块（vault-file-ops / vault-frontmatter / vault-markdown-builder），新增 30+ 数据操作方法
- 集中式 EventBus 支持 12 种类型安全事件，替代零散事件监听
- main.ts 从 251 行精简至 93 行，初始化逻辑提取到 plugin-bootstrap.ts
- UI 组件化 + 5 Tab 导航（炼丹 / 投注 / 种子 / 修炼 / 统计）

#### M4 · 服食投注系统
- 项目投注（短期增益）：项目 CRUD、丹药投注获临时增益（稀有度 +1，按品级持续 1-7 天）
- 模板投注（中期复利）：模板 CRUD、投注后守时引效果倍增、投注等级成长
- 角色投注（长期成长）：角色 CRUD、丹药投注获 XP、等级提升
- 三栏卡片布局 + 投注弹窗（丹药选择 → 效果预览 → 确认）
- 投注历史记录

#### M5 · 播种层
- 快速播种命令 `quick-seed`（命令面板一键记录）
- 种子池管理：创建 / 转化为任务 / 丢弃
- 种子 Tab 含内联输入框、转化/丢弃操作、过期提醒
- 目标管理：创建目标、进度追踪、完成统计

#### 3.1 外丹体系深化
- 药材陈化机制（fresh → aging → mature → expired 四阶段）
- 配伍相冲/冲和（性味冲突减分，40%-60% 比例触发冲和产出稀有丹药）
- 丹炉养成系统（10 级，提供品级加成/纯度下限/丹纹概率）
- 丹纹系统（宝品 10%+、神品 30%+ 概率，common/rare/legendary 三档）
- 多周期丹炉（周炉/月炉/项目炉）

#### 3.2 内丹体系
- 五境界：练气 → 筑基 → 金丹 → 元婴 → 化神，每境界 9 层
- 经脉系统：领域标签映射为经脉，10 级进度，提供采集加成
- 境界突破机制：5 种条件检测、成功率 = 70% + 心境值 × 5%、仪式感弹窗
- 封炉自动累加修炼 XP

#### 统计视图系统
- Apache ECharts 按需集成 + ChartRenderer 抽象接口（可替换）
- 活跃热力图（日历式）、领域雷达图、趋势折线图、品级饼图、番茄钟统计
- 周期选择器（近一周/近一月/近一季/近一年/全部）
- 总览卡片（总药材/总丹药/活跃天数/连续天数）

#### 集成与基础设施
- Settings 扩展：种子提醒天数、药材有效期、突破难度、默认统计周期
- 11 个命令注册（含 open-investment / attempt-breakthrough / manage-multi-furnace / open-stats / quick-seed 等）
- CSS 东方美学统一（暗色适配）、版本升至 0.2.0

## 二、近期路线

> M4（服食投注）与 M5（播种层）已于 v0.2.0 完成，外丹体系（3.1）与内丹体系（3.2）亦已落地。近期路线指向体系的进一步扩展。

### 3.3 炼器体系

> 工欲善其事，必先利其器。将工具与模板抽象为修炼法器。

**法器与灵器**
- 将用户常用的"工具"（如特定模板、工作流）抽象为法器
- 法器通过投注强化，等级提升后增强效果
- 如：一个"每日复盘"模板被炼化为法器后，使用它完成任务时额外产出药引

**装备系统**
- 玩家可"装备"有限数量的法器（如同时装备 3 件）
- 装备的法器影响采集效率和炼丹品质
- 设计意图：让用户主动思考"当前阶段最需要强化哪些习惯/工具"

### 3.4 修炼功法体系

**日常功法**
- 将重复任务模板绑定为"功法"
- 每日按时执行功法 → 积累修炼进度 → 提升内丹境界
- 设计意图：为日常习惯赋予修炼叙事，增强坚持动力

**闭关模式**
- 用户可主动进入"闭关"状态（深度工作周期）
- 闭关期间所有采集效率大幅提升，但要求更高的专注度
- 提前退出闭关会有惩罚（如心火值飙升）
- 设计意图：为深度工作提供仪式感框架和额外激励

**悟道机制**
- 系统检测到突破性创造产出时（如异常高的药材量或罕见的多领域协同），触发"悟道"
- 悟道提供一次性的大量经验值或稀有丹纹奖励
- 设计意图：回馈那些超出常规的创造性努力

## 四、远期愿景

### 4.1 灵活的属性向量系统

> "游戏中的数据不能是太过于固定的……更应该是某种复杂的向量结构，每个属性都由玩家来定义，但我们要把它接纳到游戏的玩法中。"

- 属性不再由开发者预设（如固定的"健康/力量/智力"），而是由用户自定义维度
- 用户定义的领域标签和性味作为"原始属性"，通过映射算法投射到可游玩的游戏属性上
- 主属性采用雷达图展示，档位和维度由用户设计（提供合理默认值）
- 次要属性以简洁数值变化呈现
- 引入平滑算法：防止高质或高量的瞬时变化对主属性产生过大影响，调和即时满足与延迟满足

### 4.2 适应变化的数据系统

> "系统要适应过往数据的变动，以及数据节奏的变动。"
> "游戏要容纳非游戏规则系统带来的数值变化。"

- 支持用户补充过往信息，系统标记为历史数据（不应用当日规则）
- 容纳非系统内的重大变化（如用户手动调整属性以反映现实中的重大事件）
- 用户偶然无法使用系统时不导致数据断崖或惩罚性后果
- 规则变更时对历史数据的兼容性处理：新规则向前生效，旧数据保持原始记录

### 4.3 社交与协作元素（优先级极低，不开发）

**门派系统**
- 多用户可组建"门派"，共享部分修炼成果
- 门派任务：协作完成大型目标，产出"门派丹药"

**师徒传承**
- 经验丰富的用户可作为"师父"，向新用户传授修炼路线
- 师父的模板和配置可一键分享给徒弟

### 4.4 跨平台数据兼容
- 核心逻辑与 Obsidian 解耦，数据格式标准化
- 支持导出/导入修炼数据
- 未来可迁移至独立应用、Web 端或其他笔记平台

### 4.5 AI 辅助的个性化成长建议
- 基于历史修炼数据，AI 分析用户的行为模式和成长瓶颈
- 提供个性化建议：推荐投注方向、预测境界突破时间、识别薄弱经脉
- 严格本地优先，隐私保护——AI 分析在本地运行或需用户明确授权

## 六、设计原则备忘

贯穿所有开发阶段，以下原则不应被遗忘：

1. **行为皆有回响** —— 每一次用户行为都应转化为系统内的可见积累，包括"失败"行为
2. **灵活胜于严格** —— 系统适应用户，而非用户适应系统；流程为增强性而非强制性
3. **属性由用户定义** —— 拒绝开发者预设的固定维度，拥抱用户自定义的成长方向
4. **适应变化** —— 容纳数据节奏变化、规则变更、非系统因素的影响
5. **仪式感与克制** —— 关键节点（开炉、突破）给予充分仪式感，日常交互保持克制简洁
6. **本地优先，隐私至上** —— 所有数据本地存储，网络功能需明确授权
