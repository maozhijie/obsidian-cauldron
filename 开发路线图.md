# 丹道修炼系统 · 开发路线图

> 道生一，一生二，二生三，三生万物。
> 系统亦如修行——从最初的一炉丹火，渐次演化为完整的修仙世界。

---

## 一、已完成回顾：v0.1.0（M1-M3 核心闭环）

v0.1.0 实现了从"采药"到"开炉"的最小可用闭环，完成了以下功能：

### M1 · 核心数据采集
- 领域标签系统：创建、编辑、删除，每个标签绑定性味（神识/根骨/灵动/融合）
- Todo 完成事件监听与标签解析
- 药材采集：数量折算、稀有度判定（良材/珍材）、按日存储
- 药引采集：守时引/延时引（基于任务截止时间判定）

### M2 · 番茄钟集成
- 内置简易番茄钟计时器
- 番茄钟完成 → 专注引采集
- 番茄钟中断 → 断念引采集

### M3 · 封炉与炼丹
- 每日周期管理（用户自定义封炉时间）
- 君药/臣药判定算法
- 品级计算（凡品/灵品/宝品/神品）与纯度计算
- 丹药自动命名与记录生成
- 开炉界面展示（丹药详情、关键数据一览）

## 二、近期路线

### M4 · 服食投注系统

> 炼丹非终点，服食方得道。丹药的价值在于被"投注"于成长方向。

**项目投注（短期增益）**
- 项目 CRUD 管理界面
- 将丹药投注于活跃项目，获得临时增益（药材稀有度 +1 级，持续数日）
- 投注效果可视化与到期提醒

**模板投注（中期复利）**
- 重复任务模板管理（如"每日复盘"、"周计划"）
- 投注后该模板任务的药引品质提升（守时引效果倍增）
- 投注等级随累计投注次数成长

**角色投注（长期成长）**
- 自定义角色维度（如"创作者"、"学习者"、"管理者"）
- 丹药投注后角色获得经验值，经验值满后提升等级
- 角色等级与经验值可视化展示

**设计意图：** 三个投注方向分别对应短/中/长期反馈，让玩家的每一颗丹药都有归宿，每一次选择都承载意义。

### M5 · 播种层

> 万物始于种子。一念起，便是一粒灵种入土。

**快速记录（播种）**
- 命令面板 / 侧边栏快捷入口，一键记录想法
- 想法自动创建为"种子任务"，零摩擦捕获灵感

**种子池**
- 独立视图展示所有种子任务
- 支持一键转换为正式任务（补充标签、截止时间等）
- 支持直接删除或保留为待定种子
- 定期提醒处理积压种子（可配置频率）

**项目与目标关联**
- 任务可通过标签或手动选择关联到项目
- 目标管理界面：设定目标值，药材产出自动累加进度
- 关联为增强性功能，不强制——兼容极简使用方式

**设计意图：** 播种层是"入口"而非"门槛"，降低行动启动成本，同时为后续的采药-炼丹链路提供更丰富的原料来源。

## 三、中期路线：修仙体系扩展

> 从一炉丹火，到一方洞天。系统将从"炼丹"扩展为综合修仙模拟。

### 3.1 外丹体系深化

**药材有效期与陈化机制**
- 药材不再永久有效，超期未用的药材逐渐"陈化"
- 部分药材陈化后品质下降，部分特殊药材反而因陈化增值（如陈年灵芝）
- 设计意图：营造紧迫感与策略性，鼓励及时炼化，同时为"囤药"策略留下空间

**相冲与冲和配伍**
- 不同性味的药材之间存在相冲关系（如神识与根骨相冲）
- 相冲药材同炉时降低品级，但引入"冲和"机制——特定比例的相冲反而产出稀有丹药
- 设计意图：增加配伍深度，让"今天做什么"有了更多策略考量

**丹炉养成系统**
- 丹炉本身可升级，通过持续使用积累"炉灵"
- 高等级丹炉提供额外加成（品级上限提升、纯度保底等）
- 设计意图：为长期坚持提供另一层成长反馈

**丹纹系统**
- 高品级丹药偶现"丹纹"——随机附加的特殊效果
- 丹纹类型与君药领域、药引组合相关
- 设计意图：引入惊喜感和收集欲，让每日炼丹不再完全可预测

**多周期丹炉**
- 在日炉基础上扩展：周炉（七日长炼）、月炉（三十日大炼）、项目炉（跟随项目周期）
- 长周期丹炉产出更高品级丹药，但需要更稳定的投入
- 设计意图：适配不同节奏的生产力模式，周炉适合周计划者，项目炉适合项目制工作

### 3.2 内丹体系

> 外丹养身，内丹修心。内丹体系将行为积累转化为可感知的"修炼境界"。

**修炼等级**
- 设定修炼阶段：练气 → 筑基 → 金丹 → 元婴 → 化神 → ...
- 等级提升需要累计投注特定数量和品级的丹药
- 每个境界解锁新的系统能力（如筑基后可使用周炉，金丹后可自定义丹纹等）

**经脉系统**
- 将领域标签映射为"经脉"，每条经脉对应一个成长通道
- 持续在某领域投入 → 该经脉逐渐贯通 → 获得该领域的专属加成
- 经脉可视化为人体经络图或修仙灵脉图

**境界突破机制**
- 境界之间设置"瓶颈"，需要满足特定条件方可突破
- 突破条件可包括：连续天数坚持、特定品级丹药、多领域均衡发展等
- 突破成功有仪式感展示，失败则积累"心境"值用于下次尝试

### 3.3 炼器体系

> 工欲善其事，必先利其器。将工具与模板抽象为修炼法器。

**法器与灵器**
- 将用户常用的"工具"（如特定模板、工作流）抽象为法器
- 法器通过投注强化，等级提升后增强效果
- 如：一个"每日复盘"模板被炼化为法器后，使用它完成任务时额外产出药引

**装备系统**
- 玩家可"装备"有限数量的法器（如同时装备 3 件）
- 装备的法器影响采集效率和炼丹品质
- 设计意图：让用户主动思考"当前阶段最需要强化哪些习惯/工具"

### 3.4 修炼功法体系

**日常功法**
- 将重复任务模板绑定为"功法"
- 每日按时执行功法 → 积累修炼进度 → 提升内丹境界
- 设计意图：为日常习惯赋予修炼叙事，增强坚持动力

**闭关模式**
- 用户可主动进入"闭关"状态（深度工作周期）
- 闭关期间所有采集效率大幅提升，但要求更高的专注度
- 提前退出闭关会有惩罚（如心火值飙升）
- 设计意图：为深度工作提供仪式感框架和额外激励

**悟道机制**
- 系统检测到突破性创造产出时（如异常高的药材量或罕见的多领域协同），触发"悟道"
- 悟道提供一次性的大量经验值或稀有丹纹奖励
- 设计意图：回馈那些超出常规的创造性努力

## 四、远期愿景

### 4.1 灵活的属性向量系统

> "游戏中的数据不能是太过于固定的……更应该是某种复杂的向量结构，每个属性都由玩家来定义，但我们要把它接纳到游戏的玩法中。"

- 属性不再由开发者预设（如固定的"健康/力量/智力"），而是由用户自定义维度
- 用户定义的领域标签和性味作为"原始属性"，通过映射算法投射到可游玩的游戏属性上
- 主属性采用雷达图展示，档位和维度由用户设计（提供合理默认值）
- 次要属性以简洁数值变化呈现
- 引入平滑算法：防止高质或高量的瞬时变化对主属性产生过大影响，调和即时满足与延迟满足

### 4.2 适应变化的数据系统

> "系统要适应过往数据的变动，以及数据节奏的变动。"
> "游戏要容纳非游戏规则系统带来的数值变化。"

- 支持用户补充过往信息，系统标记为历史数据（不应用当日规则）
- 容纳非系统内的重大变化（如用户手动调整属性以反映现实中的重大事件）
- 用户偶然无法使用系统时不导致数据断崖或惩罚性后果
- 规则变更时对历史数据的兼容性处理：新规则向前生效，旧数据保持原始记录

### 4.3 社交与协作元素（优先级极低，不开发）

**门派系统**
- 多用户可组建"门派"，共享部分修炼成果
- 门派任务：协作完成大型目标，产出"门派丹药"

**师徒传承**
- 经验丰富的用户可作为"师父"，向新用户传授修炼路线
- 师父的模板和配置可一键分享给徒弟

### 4.4 跨平台数据兼容
- 核心逻辑与 Obsidian 解耦，数据格式标准化
- 支持导出/导入修炼数据
- 未来可迁移至独立应用、Web 端或其他笔记平台

### 4.5 AI 辅助的个性化成长建议
- 基于历史修炼数据，AI 分析用户的行为模式和成长瓶颈
- 提供个性化建议：推荐投注方向、预测境界突破时间、识别薄弱经脉
- 严格本地优先，隐私保护——AI 分析在本地运行或需用户明确授权

## 五、技术演进方向

### 5.1 架构解耦
- 核心逻辑（采集引擎、配伍算法、投注系统）抽离为独立模块，不依赖 Obsidian API
- UI 层作为适配器对接 Obsidian，便于未来迁移到其他平台
- 数据层抽象：统一的数据访问接口，底层可切换存储方式

### 5.2 代码质量提升
- 引入 ESLint 严格规则集，统一代码风格
- TypeScript `strict: true` 全面覆盖
- 模块化拆分：单文件不超过 200-300 行，职责单一

### 5.3 单元测试覆盖
- 核心算法（配伍、品级计算、纯度计算）优先覆盖单元测试
- 数据持久化层的读写测试
- 周期管理的边界条件测试（跨日、补封炉、时区等）

### 5.4 性能优化
- 丹药列表虚拟滚动（数据量增长后的长列表性能）
- 药材/药引采集增量写入，避免全量重写
- 封炉计算后台执行，不阻塞 UI
- 数据文件按日归档，控制单文件体积

### 5.5 插件 API 开放
- 定义开放 API，允许第三方扩展：
  - 自定义采集源（如集成其他番茄钟插件、时间追踪工具）
  - 自定义炼丹规则（新的配伍逻辑、品级算法）
  - 自定义投注效果
- 事件总线：对外暴露关键事件（药材采集、封炉完成、丹药产出），供其他插件监听

## 六、设计原则备忘

贯穿所有开发阶段，以下原则不应被遗忘：

1. **行为皆有回响** —— 每一次用户行为都应转化为系统内的可见积累，包括"失败"行为
2. **灵活胜于严格** —— 系统适应用户，而非用户适应系统；流程为增强性而非强制性
3. **属性由用户定义** —— 拒绝开发者预设的固定维度，拥抱用户自定义的成长方向
4. **适应变化** —— 容纳数据节奏变化、规则变更、非系统因素的影响
5. **仪式感与克制** —— 关键节点（开炉、突破）给予充分仪式感，日常交互保持克制简洁
6. **本地优先，隐私至上** —— 所有数据本地存储，网络功能需明确授权
